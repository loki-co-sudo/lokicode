# lokicode

OpenRouter 経由で AI チャットを使える、VS Code 風のデスクトップコードエディタ。
Tauri v2 + React + TypeScript + Vite + Monaco + Tailwind CSS。

## 開発

```bash
npm run tauri dev     # デスクトップアプリ（要 Rust）
npm run dev           # フロントエンドのみ（ブラウザ）
```

API キーは `.env`（`.gitignore` 済み）の `VITE_OPENROUTER_API_KEY` から読み込む。
`.env.example` には実キーを入れないこと（プレースホルダのみ）。

## 開発ルール

- **機能や挙動を変更したら、毎回 `README.md` も同じ変更で更新する**（コミット前に。日本語で）。README を「完了の定義」に含めること。

### ループ開発（すべてのコーディングタスクに適用）

コーディングは1回きりの「直線」ではなく、チェックが通るまで回す「ループ」で行う:

1. 【変更】コードを修正・追加する。
2. 【検証】`npm test` と `npm run build` を実行する（Rust を触ったら `cargo clippy` も）。
3. 【修正】失敗したらエラーログを読み、原因を特定して直し、手順2へ戻る。

停止条件:
- 【成功】すべて通過したときだけ「完了」と報告する。**通過した出力（末尾）を証拠として添える**。
- 【上限】ループは最大 **5 回**。使い切ったら止めて、何が残っているかを報告する。
- 【手詰まり】**同じエラーが2回連続**したら思考の固着と判断して止め、別コンテキスト
  （新しいセッション/会話）での修復を提案する。

絶対禁止:
- チェック出力（通過の証拠）なしで「完了」と報告すること。
- テストを通すためにアサーションを削除・弱体化・スキップすること。直すのはコードであり、評価基準ではない。

## コミット規約

- コミットメッセージは **English のプレフィックス**（`feat:` / `fix:` / `chore:` / `docs:` / `refactor:` など）+ **日本語の説明**で書く。
  - 例: `feat: AIチャットのストリーミング応答に対応`

# 開発ガイド（AI エージェント向け・必読）

このセクションは、**どのレベルのモデルでも**この repo の AI 機能を安全に改善し続けられる

## 1. どこに何があるか（アーキテクチャ地図）

## 2. エージェント効率の設計原則（この5つに沿って変更する）

1. **適応的計算**: 計算量（API 呼び出し・反復回数）は課題の難しさに合わせて増減させる。
   固定の「常に N 回」を追加してはいけない。実装手段は
   (a) 難易度ルーター、(b) 検証器のスコアによる早期終了（合格点に達したら止める）、
   (c) エフォートプリセット（`EFFORT_PARAMS`＝速度/バランス/品質。ユーザーが
   コスト・速度・精度のトレードオフを調整する唯一の一括ノブ）。新しい反復処理を
   足すときは「いつ止まるか」「易しい課題ではスキップされるか」を必ず設計する。
2. **コスト効率ルーティング**: 量をこなす仕事（調査・ドラフト・機械的なツール操作）は
   **安価な thinking モデル**、テコが効く要所（設計ブリーフ・検証の採点・最終合成）は
   **強い synthesis モデル**に割り当てる。新フェーズを足すときは「これはどちらか」を決める。
   **検証器（JUDGE）は必ず強モデルで実行**（弱い judge は誤りを見逃す＝安全網に穴）。
3. **接地（グラウンディング）**: モデルに事実を主張させる前に、**実ファイルを読ませる**。
   証拠ゼロで生成する経路を作らない（読み取り専用ツールでの調査を最低1回挟む）。
   出力には `file:line` の引用を要求する。
4. **検証の分離**: 「できました」という自己申告を信用しない。品質判定は
   **独立した検証器（別呼び出しの LLM-as-judge、ルーブリック採点）**か
   **実行結果（ビルド/テストの exit code）**で行う。
5. **並列化と収穫逓減**: 互いに独立な呼び出しは `Promise.all` で並列に（体感遅延は最遅の
   1本分になる）。並列サンプル数は **1→2 の改善が最大で 3 以降は逓減**するので、
   幅を増やす変更はエフォート「品質」限定にする。

## 3. よくある変更のレシピ（手順どおりにやれば安全）

**チューニングノブを追加・変更するとき**
1. `specs/` に仕様（動機・値の根拠・検証方法）を書く（新規なら新ファイル）。
2. `agentSettings.ts` に getter/setter と既定値（localStorage キーは `lokicode.*`）。
3. 使う側（`reasoning.ts` など）は**実行のたびに getter を読む**（定数に焼き込まない）。
4. 呼び出し回数が変わるなら `cost.ts` の `pipelineShape` を**同じ分岐で**更新。
5. `cost.test.ts` / `agentSettings.test.ts` にテスト追加 → `npm test`。
6. UI（ChatPane）に露出するならツールチップで「何とトレードオフか」を書く。
7. README の該当節を更新（下記 5.）。

**システムプロンプト（エージェント命令）を変えるとき**
- 命令は「〜しない」だけでなく「（理由）だから（具体的動作）」の形で書く。
- 追加する前に既存の節と矛盾しないか全文を読み直す。重複は削る（長文化＝毎回のコスト増）。
- 効果確認は試走で: Agent＋ディープ ON・広さ3・深さ3〜4 で実行し、F12 コンソールの
  `[deepthink]`（フェーズ別秒数・呼び出し回数）と `[agent]`/`[execute]`（反復ごと）を見る。

**ディープシンクのフェーズを追加するとき**
- そのフェーズは thinking / synthesis のどちらで走るか（原則2）、いつスキップされるか
  （原則1）、`pipelineShape` への反映（コスト概算の整合）、`phaseTag` の設定
  （タイミング計測）の4点を必ず揃える。

## 4. してはいけないこと（過去の教訓。詳細は specs/ に記録済み）

- 旧 v1 の線形ループ・巻き戻し機構を復活させない（0.40.0 で削除済み・戻さない方針）。
- `CONVERGED` のような**自己申告の収束判定**を導入しない（検証器か実行結果で判定する）。
- 検証器（JUDGE）を安価モデルに下げない。
- アンサンブル（複数ドラフト）を**証拠なし**で走らせない（最小接地を外さない）。
- `pipelineShape` を更新せずに呼び出し回数を変えない（コスト概算と校正が狂う）。
- OpenRouter のルーター系メタモデル（`openrouter/fusion` / `auto` / `pareto-code`）を
  thinking/synthesis に推奨しない（価格 -1＝変動、挙動が特殊）。

## 5. 完了の定義（毎回のチェックリスト）

1. `npm test` が通る（純ロジックのユニットテスト）。
2. `npm run build` が通る（tsc 型チェック込み）。
3. 機能・挙動を変えたら **同じコミットで README.md を更新**（日本語）。
4. 非自明な機能は specs/ に仕様を書いてから実装する。
5. コミットは English prefix + 日本語説明。リリースは README の「リリース手順」どおり
   （`package.json` / `tauri.conf.json` / `Cargo.toml`（+ `Cargo.lock`）の version を
   上げ、`vX.Y.Z` タグを push → GitHub Actions が公開）。
