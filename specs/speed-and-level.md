# 速度改善（フェーズ反復バジェット）と実効レベルの事前推定 — 設計仕様

対象バージョン: 1.5.0
関連: [effort-presets.md](effort-presets.md) / [model-gate.md](model-gate.md) / e2e 実測（1.4.1）

## 1. 動機: 実測で判明したボトルネック

1.4.1 の実走 e2e（DeepSeek v4 pro/flash）の実測:
- balanced の分析ラン: **約21分**。quality: 約24分。
- 主因は**調査・改善フェーズのミニループ**。軽量モデル（flash）は1ターンに1ツール呼び出し
  しか行わない傾向があり、上限は `getMaxIterations()`（既定 **50**）をそのまま共有。
  e2e では 20 反復・プロンプト 35k トークンまで膨張（反復のたびに全会話を再送＝
  遅延とコストの両方が二乗的に悪化）。
- classify（実行要否）→ brief（設計）が**直列**だが相互依存なし（各 2〜15s の無駄）。

## 2. 速度改善の仕様

### 2.1 フェーズ反復バジェット（最重要）
- `AgentOptions.maxIterations?: number` を追加（未指定なら従来どおり設定値）。
- `EFFORT_PARAMS` に `phaseIterations` を追加: **speed 6 / balanced 10 / quality 14**。
  ディープシンクの**読み取り専用フェーズ**（調査・接地・改善・非アンサンブルの
  ドラフト/最終）のツールループに適用。**実行フェーズは対象外**（本作業は従来上限）。
- 根拠: e2e 実測で調査に必要な読みは数ファイル。20 反復の大半は「1ターン1読み」の
  逐次徘徊で、情報量は 6〜8 反復で飽和していた（それ以降は同じファイルの再読が中心）。

### 2.2 調査プロンプトのバッチ指示
INVESTIGATOR に「独立した読み取りは**1ターンに複数ツール呼び出し**でまとめる」
「ツールターンは最大 N 回（バジェット）——答えられる材料が揃ったら即座に構造化出力へ」
を明記。トップレベルのエージェント指示にはあるが、調査フェーズの system には
無かった（効率指示の欠落）。

### 2.3 classify ∥ brief の並列化
NEEDS_EXEC 分類と設計ブリーフは相互依存が無いので `Promise.all` で並列に。
（呼び出し回数は不変 → `pipelineShape` 変更なし。反復バジェットも構造呼び出し数を
変えない＝実呼び出し数の減少は toolMult 校正が自然に学習する。）

## 3. 実効レベルの事前推定（モデル組み合わせ → どの級の知能で動くか）

### 3.1 考え方（正直なヒューリスティック）
テスト時計算のスケーリング研究の一貫した結果:
- 検証器ガイドの反復・best-of-N・自己整合性で、**単発比 おおむね「1ティア上」まで**の
  実効性能向上が得られる（それ以上は逓減が支配）。AA 知能指数でおおむね **+10前後が上限**。
- 最終品質は**合成モデル（最終合成・検証器）が支配**し、証拠品質（思考モデル）が下限を作る。

### 3.2 推定式（lib/modelGate.ts・純関数）
```
base = 合成モデルの知能指数（未公開なら推定不能と表示）
uplift =
  +2  接地調査あり（ツール ON）
  +1.5 × min(depth, 3)  検証×改善ラウンド（3 で飽和）
  +2  アンサンブル有効（幅2） / +3（幅3）
  +1  judge 多数決（judgeSamples ≥ 2）
  +1  広さ ≥ 2（独立観点の調査）
cap: uplift ≤ 12
証拠ペナルティ: 思考モデルの指数が 25 未満なら uplift を 1/2（弱い証拠が天井）
effective = round(base + uplift)
```
表示は「単発の合成モデル ≈ base → パイプライン実効 ≈ effective」の形で、
**推定であることを明記**（ツールチップに根拠と限界を記載）。

### 3.3 UI
ディープシンク ON 時、コスト概算の行に併記:
`🧠 実効レベル(推定): 指数 ~62（合成 55 ＋深化 +7）`
指数未公開のモデルでは「推定不能（知能指数が未公開）」。

### 3.4 正直さ
- これは**目安**であり、ベンチマーク実測ではない。over-claim しない文言にする。
- Fable 5 級（指数 70+）には、base が 60 級でも届かない場合がある——その事実は
  隠さず表示する（「+12 が上限」という飽和を式に含める）。
