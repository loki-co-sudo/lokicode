# 推論エフォート（速度 / バランス / 品質）と最小接地 — 設計仕様

対象バージョン: 1.1.0
関連: [deep-reasoning-v2.md](deep-reasoning-v2.md) / [agent-v3.md](agent-v3.md)

## 1. 動機

lokicode の推論パイプライン（Agent / ディープシンク）には品質を上げるノブが増えたが、
**コスト・速度・精度のトレードオフを1箇所で選ぶ手段がなかった**。
検証合格点（PASS_SCORE）・アンサンブル幅（ENSEMBLE_SAMPLES）・十分性ゲートの回数は
コード内の定数で、ユーザーが「今日は速く/安く」「この課題は品質最優先」と切り替えられない。

最新の推論時スケーリング研究の一致した知見は次のとおり:

- **テスト時計算のスケーリング**（Snell et al. 2024, "Scaling LLM Test-Time Compute Optimally"）:
  同じモデルでも検証器ガイド付きのサンプル数・改善回数を増やすほど精度が上がるが、
  **収穫逓減**があり、課題難易度に応じて計算量を配分するのが最適。
- **自己整合性 / best-of-N**（Wang et al. 2022; Lightman et al. 2023）: 並列サンプル＋選抜は
  誤りの相関が低いほど効く。幅 1→2 の伸びが最大で、3 以降は逓減。
- **Mixture-of-Agents**（Wang et al. 2024）: 複数ドラフト→統合は廉価モデルでも
  フロンティア級に迫るが、呼び出し回数は幅に比例する。
- **LLM-as-judge の閾値 = 適応的 halting**（deep-reasoning-v2 §Tier1-B）: 合格点を上げる＝
  平均反復回数（コスト・時間）を増やして精度を買う、そのままの調整レバー。

つまり「1 つの最適点」は存在せず、**プリセットとしてユーザーに露出するのが正しい**。

## 2. 仕様: EffortLevel

`lib/agentSettings.ts` に追加。localStorage キー `lokicode.effort`、既定 `balanced`。

| パラメータ | speed（速度） | balanced（既定） | quality（品質） |
| --- | --- | --- | --- |
| passScore（検証の合格点・早期終了） | 78 | 85 | 92 |
| escalateBelow（強モデル昇格の閾値） | 60 | 70 | 78 |
| ensembleSamples（MoA/best-of-N の幅） | 1（=無効） | 2 | 3 |
| sufficiencyRounds（十分性ゲート最大回数） | 1 | 2 | 3 |

- 幅・閾値は §1 の「1→2 が最大の伸び、3 で逓減」に沿って設定。
- `ensembleSamples: 1` はアンサンブル自体を無効化（トグルより優先して安く・速く）。
- 既存の「アンサンブル」トグル・深さ/広さスライダーはそのまま**微調整**手段として残す。
  エフォートは「粗い一括プリセット」、スライダーは「細かい上書き」という2層。

## 3. 仕様: エージェント指示のエフォート連動

`buildSystemPrompt`（ChatPane）に、エフォート別の運転指示を1ブロック追加する。
根拠: エージェント系プロンプティングの実務知見（persistence 節・コンテキスト収集の
早期打ち切り基準・検証の深さの明示は、往復数と回答品質の主要な操作点）。

- 共通（全レベル）:
  - **persistence**: 依頼が解決するまでターンを終えない。作業を求められたら
    「やり方の説明」で止まらず実行する。
  - **早期打ち切り基準**: 変更すべきファイル・箇所を特定できた時点で調査をやめて着手する
    （調査のしすぎはコストと遅延の主因）。
- speed: 追加の裏取り読み・任意の検証を省き、最短手数で完了。回答は要点のみ。
- balanced: 現行の既定（変更が失敗しうるときだけ検証）。
- quality: 実質的な変更のたびに裏取り（変更箇所の再読・安価ならビルド/テスト）、
  終了前に要件を1つずつ照合。

## 4. 仕様: ディープシンクの最小接地（accuracy fix）

現状のバグに近い挙動: `breadth == 1` のとき調査フェーズが走らず、かつアンサンブル有効時の
ドラフトは**ツールなしの並列プレーン生成**になるため、**証拠ゼロのままドラフト（＝幻覚温床）**
になる経路が存在する（`depth >= 3 && samples == 1 && useTools` で発生）。

対策: **アンサンブル有効・ツールあり・証拠なし**のとき、ドラフト前に GOAL を副問とした
読み取り専用の接地調査を **1 回**だけ実行して evidence を作る（+1 ループ呼び出し）。
Reflexion / 接地生成の知見どおり、「生成前に1度でも実物を読む」ことが無根拠クレームを
大きく減らす。コスト増は調査1回ぶんで、アンサンブル（≥5呼び出し）に比べ小さい。

## 5. コスト概算との整合

`cost.ts` の `pipelineShape` は唯一の真実源なので、同じ入力を受ける:

- `ensembleSamples`（エフォート由来）を引数に追加（既定 2 で後方互換）。
- アンサンブル判定を `ensembleSamples > 1 && (breadth > 1 || depth >= 3)` に更新。
- 最小接地: `useTools && ensemble && breadth == 1` のとき `invest: 1`。

## 6. 実装しないこと（見送り）

- 検証器の多数決（judge ensembling）: 強モデル judge ×N はコスト増が大きく、
  まず閾値プリセットで様子を見る。
- 難易度ルーターの多段化（易/中/難でエフォート自動選択）: ルーター1コールの分類精度が
  未検証のため次版候補。現行の routedHard（deep 判定で深さ/広さを底上げ）は維持。
- 多腕バンディットによるモデル自動選択: 台帳（校正データ）の蓄積が先。

## 7. 検証方法

- `npm test`: EFFORT_PARAMS の写像・pipelineShape の新分岐のユニットテスト。
- 手動試走: Agent＋ディープ ON、`depth=3, samples=1` で「証拠なしドラフト」経路が
  接地調査を1回挟むこと（F12 の `[deepthink] investigate` ログ）を確認。
- エフォート speed で MoA が無効化され呼び出し回数が減ること（`[deepthink] TOTAL` の calls）。
