# 難易度ルーター連動と検証器の多数決 — 設計仕様（ディープシンク beta 卒業）

対象バージョン: 1.4.0
関連: [effort-presets.md](effort-presets.md) / [model-gate.md](model-gate.md) / [deep-reasoning-v2.md](deep-reasoning-v2.md)

## 1. 動機: beta 表記を外すために残っていたもの

v1.0 でディープシンクを beta とした実質的な理由は「品質が不安定」だったこと。
その主因は以下で、v1.1〜1.3 までにほぼ潰した:

| 不安定要因 | 対策（実装済み） |
| --- | --- |
| 検証器が賢さの壁の下（無料ルーター） | 強モデル JUDGE（0.42.0）＋モデル要件ゲート（1.3.0） |
| 証拠ゼロのドラフト経路（幻覚温床） | 最小接地（1.1.0） |
| コスト/速度/精度の調整手段がない | エフォートプリセット（1.1.0） |
| 意図から逸れた回答 | 設計ブリーフ GOAL/CRITERIA 固定（0.23.0） |

残っていた高ROI項目が本仕様の2つ。これを実装した時点で、未実装のロードマップは
「増分的な最適化」（バンディットのモデル自動選択・証拠の構造化・中断即応）のみとなり、
**beta（＝中核が実験段階）表記を外す**のが正確な状態になる。

## 2. 難易度ルーター連動（オート時のディープ設定自動化）

現状: オート ON で deep 判定されると一律 `depth≥4 / samples≥3 / ensemble=true` に底上げ。
「深い」の中の難易度差（普通に難しい vs 極めて難しい）を区別せず、過剰/過小になる。

仕様:
- ルーターの分類を4値に拡張: `trivial / agent / deep / deep-hard`。
  - `deep`: 多段調査・設計・慎重な検証が要る難課題。
  - `deep-hard`: 極めて難しい/リスクが高い課題（多ファイル横断のアーキテクチャ分析、
    原因不明の微妙なバグ、大規模で危険なリファクタ）。
- オート ON 時のディープ設定（スライダーは下限として尊重、上書きは底上げのみ）:
  - `deep` → depth ≥ 3、samples ≥ 2、アンサンブルはユーザー設定を維持
  - `deep-hard` → depth ≥ 5、samples ≥ 3、アンサンブル強制 ON
    （エフォート「速度」の幅1はそのまま優先＝ユーザーの一括ノブが最上位）
- 応答の解析は純関数 `parseRoute(content)` に分離してユニットテスト（`deep-hard` を
  `deep` より先に照合。表記揺れ deep_hard/deephard を許容）。誤判定・API 失敗時は
  従来どおり `agent` にフォールバック。

## 3. 検証器の多数決（品質エフォート限定）

LLM-as-judge は1サンプルだと採点ノイズで早期終了/昇格の判断が揺れる。
自己整合性（self-consistency）の知見どおり並列サンプルで平滑化するが、
コストが線形に増えるため**エフォート「品質」限定**（適応的計算の原則）。

- `EFFORT_PARAMS` に `judgeSamples` を追加: speed 1 / balanced 1 / quality 2。
- 検証フェーズは judge を `judgeSamples` 本**並列**実行し、
  - スコア = **min**（保守的: 甘い judge に引きずられて見逃さない）
  - 欠陥リスト = 和集合（重複除去）
- `cost.ts` の `pipelineShape` に `judgeSamples` を追加（judge = depth × judgeSamples）。
