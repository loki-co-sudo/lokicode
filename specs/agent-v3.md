## 追加する3コンポーネント

### 1. 難易度ルーター（入口ゲートウェイ）
- **目的**：易しい依頼に重い処理を使わない＝速度・コスト最適化。
- **方式**：送信時に超軽量1コールで `trivial | agent | deep` を分類し、実行経路を自動選択。
  - trivial → プレーン応答（ツール無し）
  - agent → ツール使用エージェント
  - deep → ディープ推論パイプライン
- **UI**：「オート（自動振り分け）」トグル（既定 OFF）。ON のときだけ分類が走り、手動トグルを上書き。選択結果はチャットに小さく表示。
- **実装**：`lib/router.ts` の `classifyTask(text, model)`（厳格プロンプトで1語返し、解析失敗時は agent にフォールバック）。ChatPane の handleSend で `useDeep/useAgent` を分類結果から決定。

### 2. 賢さの壁の担保（強い検証器）
- **目的**：検証役を壁の下に置かない。無料/弱い思考モデルでも誤りを捕まえる。
- **方式**：**JUDGE（検証採点）を synthesis（強）モデルで実行**。調査・ドラフト・改善は従来どおり thinking（安）。
  - 批判役だけ強モデルにすることで、量は安いまま「最後の砦」が壁を超える。
  - コスト：検証回数ぶん強モデル呼び出しが増える（cost.ts の概算も judge を強モデル側に計上）。
- 補足：将来、モデルの知能指数が取れる場合は thinking が壁の下のとき自動で強検証に切替、等の発展余地。

### 3. 実行接地の自己修正（Reflexion）【実装済 0.43.0】
- **目的**：コーディングで最強のレバー。AI 変更後に**実際にビルド/テストを走らせ**、失敗を食わせて緑まで反復。
- **方式**：設定「検証コマンド（任意）」（例 `npm run build` / `npm test`、agentSettings.ts `getVerifyCommand`/`MAX_VERIFY_ATTEMPTS`）。**Agent モードで AI がファイルを変更した場合**、ターン後にワークスペース直下でコマンドを実行し、**非0終了なら出力（最大6000字にクリップ）を添えて修正を依頼→再実行**（最大 `MAX_VERIFY_ATTEMPTS`=2 回）。タイムアウトは既存設定を流用。実装は ChatPane handleSend の agent 分岐（agentCb/agentOpts を fix 実行でも再利用）。
- 既定 OFF（コマンド未設定なら何もしない）。ディープ推論はファイルを書かないため対象外（Agent モードのみ）。

## 実装順とリスク
- #2（強い検証器）: 最小・最大効果・低リスク → 先行実装。
- #1（難易度ルーター）: 中規模（新規 lib＋トグル＋分岐）。速度に直結。
- #3（実行接地）: 最大・要設計（設定＋ポストターンのループ）。慎重に。

## 復元/無効化
- #1 ルーター：トグル OFF（既定）。#2 強検証器：cost が気になる場合は JUDGE を thinking に戻す1行。#3：検証コマンド未設定で無効。
