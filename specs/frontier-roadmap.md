# フロンティア接近ロードマップ — 「モデルを上げる」以外の全レバー

作成: 1.5.0 時点。対象: 今後の複数バージョン。
これは**実装可能な形で書いた改善計画**である。各項目に 動機（なぜ天井の外か）→
実装スケッチ（触るファイル・ゲート）→ 検証方法 → コスト影響 を揃えてあるので、
どのレベルのモデルでも上から順に実装できる。

## 0. 前提の整理: 「+1ティア飽和」は何に対する限界か

実効レベル推定（[speed-and-level.md](speed-and-level.md)）の「uplift は指数 +12 で飽和」は
**固定タスク・固定文脈のまま、同一モデルの出力分布から再サンプルする計算増**
（best-of-N・自己整合性・検証器ガイドの反復）に対する経験則である。
つまり飽和するのは「同じ問いを同じ材料で何度も解き直す」戦略だけ。
次の4軸は**この飽和則の適用範囲の外**にあり、ここが本ロードマップの核心:

1. **検証信号を LLM から実行に変える** — 「検証は生成より易しい」の非対称性を使う。
   テスト・コンパイラ・型検査の合否は判定側の知能に依存しない**地上真実**であり、
   実行フィルタリングは中位モデルを競プロ上位に押し上げた実績がある
   （AlphaCode の大量生成→実行選抜、CodeT のテスト駆動選抜）。
2. **タスク構造を変える（分解）** — 弱モデルが解けないのは「問題が能力圏の外」のとき。
   合成タスクを能力圏内のサブ問題に分解すれば、単発では何回サンプルしても解けない
   問題が系として解ける（Least-to-Most、factored cognition）。
3. **文脈を変える（context engineering）** — フロンティアとの差の相当部分は
   「大量のコードを正確に保持する力」。読ませる前に整理（リポジトリマップ・
   証拠の構造化・圧縮）すれば、弱モデルの読解負荷そのものを下げられる
   （aider の repo-map が実証済みの先行例）。
4. **時間で蓄積する（記憶）** — 1回の呼び出しの賢さは変わらなくても、
   検証済み知識・失敗パターンをセッション横断で蓄積すれば**系として**賢くなる
   （Reflexion の長期記憶、Voyager のスキルライブラリ）。

## 優先順位つき実装計画

### P1. 実行検証のディープシンク統合（テスト駆動生成）〔軸1・最高ROI〕
- **動機**: ディープシンクの品質判定は現状 LLM-judge のみ＝judge の知能が上限。
  実行結果（exit code）を検証器に加えれば、上限が「テストの網羅性」に変わる。
- **実装スケッチ**:
  1. 実行フェーズ（reasoning.ts の fast path）の完了後、Agent モードの検証ループ
     （ChatPane にある verify loop）と同等の処理を回す。純ロジックは `lib/loop.ts` を再利用。
     上限・手詰まり判定はループモードと同一（LOOP_MAX_ATTEMPTS / errorSignature）。
  2. 発展: コード生成系タスクでは**先にテストを生成**（thinking・安価）→ 実装 →
     テストが緑になるまで反復。「テストを弱体化しない」禁止事項を必ず添える
     （[loop-mode.md](loop-mode.md) の絶対禁止と同文）。
- **ゲート**: 検証コマンド設定済み or テスト生成が可能なコードタスクのみ。分析タスクは対象外。
- **検証**: e2e ハーネスに「わざとバグを入れた実装依頼→緑まで反復」ケースを追加。
- **コスト影響**: 失敗時のみ反復（成功なら +1 コマンド実行）。pipelineShape は execute 系の
  ため構造呼び出し数は不変（toolMult が学習）。

### P2. リポジトリマップ（記号インデックス）〔軸3〕
- **動機**: 調査フェーズは毎回ゼロから grep/read で探索している。ファイル→主要シンボル
  （関数・クラス・export）の地図を先に渡せば、読む箇所の特定が1ターンで済み、
  速度と接地精度が同時に上がる。埋め込み基盤は**不要**（aider 方式: 構文ベースで十分）。
- **実装スケッチ**:
  1. Rust 側（src-tauri/src/search.rs 付近）に `repo_map` コマンド: 対象拡張子の各ファイル
     から `fn|function|class|export|pub|interface|struct` 行を正規表現抽出し
     `path: symbol行…` の一覧を返す（上限 ~8k トークン、深さ/サイズでクリップ）。
  2. reasoning.ts: ブリーフと INVESTIGATOR の system に地図を添付（breadth≥1・useTools時）。
  3. agent.ts のトップレベルにも将来展開可（まずディープで実証）。
- **検証**: e2e Run A で調査フェーズのツール反復数と所要時間が減ること（計測比較）。
- **コスト影響**: 入力トークン +数千/フェーズ ↔ ツール反復の削減で相殺見込み。実測で判断。

### P3. 解レベルの分解（SUBTASKS 合成）〔軸2〕
- **動機**: 現状の分解は**調査**だけで、ドラフト（解く工程）は1回の monolithic 生成。
  大きな実装・横断リファクタで弱モデルが崩れる主因。サブタスク単位なら能力圏内。
- **実装スケッチ**:
  1. BRIEF に `SUBTASKS:`（2〜5個・相互依存が薄い順）を追加出力させる（deep-hard か
     ルーターが deep のときのみ。trivial な依頼に分解を強いない＝適応的計算）。
  2. 各サブタスクを thinking で**個別に**解く（evidence＋該当サブタスクのみの文脈、
     phaseIterations 上限つき、並列可なら Promise.all）。
  3. synthesis が COMPOSE: サブ解を統合し、界面の不整合（命名・型・重複）を解消。
  4. 検証（judge / P1 の実行検証）は統合後の全体に対して従来どおり。
- **ゲート**: `route === deep/deep-hard` かつ BRIEF が SUBTASKS を2個以上返した場合のみ。
- **検証**: e2e に「複数ファイルにまたがる機能追加」ケースを追加し、単発ドラフト比の
  judge スコアを比較。pipelineShape に subtask 分岐を追加（コスト整合・原則）。

### P4. 検証済み証拠キャッシュ（セッション横断・ハッシュ失効）〔軸4〕
- **動機**: 毎ランで同じファイルを読み直している。VERIFIED 事実を蓄積すれば
  2回目以降のランは「既に知っている状態」から始まる＝蓄積による実効知能。
- **実装スケッチ**:
  1. `lib/evidenceCache.ts`: `{ factText, files: [{path, contentHash}] }` を localStorage
     （サイズ上限つき）に保存。調査フェーズの出力から VERIFIED 行を抽出して記録。
  2. ラン開始時、引用ファイルの**現在のハッシュが一致するものだけ**を「既知の事実
     （検証済み・日付つき）」として INVESTIGATOR/ドラフトの文脈に注入。不一致は破棄。
  3. **重要**: 回答そのもののキャッシュは絶対にしない（HSN 議論で棄却済み。
     コード変更で古い回答を返す事故になる）。キャッシュするのは**ファイル単位で
     失効管理できる原子的事実**のみ。
- **検証**: 同一質問の2回目のランで調査呼び出し数が減り、かつファイル変更後は
  古い事実が注入されないユニットテスト（ハッシュ不一致で棄却）。
- **コスト影響**: 2回目以降の invest 呼び出し減。pipelineShape は不変（概算は保守側）。

### P5. 欠陥パターンの長期記憶（Reflexion メモリ）〔軸4〕
- **動機**: judge の defect 指摘は毎回使い捨てている。繰り返し出る欠陥型
  （例: 無根拠な数値、監査ドリフト）を蓄積し、**次回のドラフト指示に事前注入**すれば
  同じ失敗を予防できる＝プロンプトが経験で強くなる。
- **実装スケッチ**: `lib/defectMemory.ts`: defect 文字列を正規化して頻度カウント
  （localStorage・上限つき）。頻度上位3件を DRAFT/FINAL 指示に「よくある失敗（避けること）」
  として添付。手動クリア UI（設定）を用意。
- **検証**: ユニットテスト（正規化・上位抽出）。試走で judge 初回スコアの平均が上がるか。

### P6. 欠陥誘導ビームサーチ（deep-hard・品質限定）〔軸1+探索〕
- **動機**: 現状の改善は線形（1本の draft を直し続ける）。局所解に嵌まると
  手詰まり。最重要欠陥に対し**複数の修正案を分岐生成**→judge 選抜（幅2の
  ビーム）は、線形改善より高い到達点を持つ（ToT/LATS。ただしコスト増）。
- **ゲート**: `route === deep-hard` かつエフォート品質のみ。幅2・1分岐まで。
- **実装スケッチ**: verify ループ内で `score < escalateBelow` が2回続いたら、
  トップ欠陥に対する2案を並列生成→judge で選抜して続行。pipelineShape に分岐追加。

### P7. バンディット・モデル自動選択〔既載・据え置き〕
- judge スコア×実コストの台帳（cost.ts の校正と同じ器）→ ε-greedy で thinking モデルを
  自動選択。台帳が十分溜まるまで着手しない（P1〜P4 が先）。

## 実装順の根拠
P1（検証の質的転換・既存部品で組める）→ P2（全フェーズの土台・速度も稼ぐ）→
P3（能力圏の拡張・P2 の地図があると分解精度が上がる）→ P4/P5（蓄積系・小さく安全）→
P6（コスト増・ゲート厳格）→ P7（データ待ち）。

## してはいけないこと（本ロードマップの文脈で再掲）
- **回答キャッシュ**は作らない（P4 は「ハッシュ失効つきの原子的事実」限定。理由は
  メモリ/HSN 議論参照: コード変更後に古い回答を返す事故）。
- 分解（P3）を trivial/agent 級の依頼に適用しない（適応的計算の原則1）。
- ビーム（P6）を無条件に広げない（幅3以上は逓減が支配、コストだけ増える）。
- 実行検証（P1）で「テストの弱体化で緑にする」抜け道を許さない（loop-mode の絶対禁止）。
