# lokicode

OpenRouter 経由で **AI エージェント**を使える、**VS Code 風のデスクトップコードエディタ**です。
単なるチャットではなく、AI が**ツールを使ってファイルの読み書きやシェルコマンド実行を行い、実際に作業を代行**します（コード生成・編集・実行など）。

**技術スタック:** Tauri v2 + React + TypeScript + Vite + Monaco Editor + Tailwind CSS
**対応プラットフォーム:** 現在は **Windows 10/11 のみ**（macOS / Linux はロードマップ）。
**ライセンス:** [MIT](LICENSE)

## セットアップ（はじめての方へ）

目的別に2通りあります。**まず使ってみたいだけなら A**（ビルド不要）、**コードをいじる・自分でビルドするなら B** です。
どちらも先に **0. 前提環境** を済ませてください。

### 0. 前提環境（必要なもの）

| 使い方 | 必要なもの |
| --- | --- |
| **A. インストールして使う** | Windows 10/11。**WebView2**（Windows 11 は標準同梱。Windows 10 は[こちら](https://developer.microsoft.com/microsoft-edge/webview2/)から）。それ以外は不要。 |
| **B. ソースから動かす** | 上記に加えて、**Node.js 18 以上**＋npm、**Rust ツールチェーン**（<https://rustup.rs>）、OS 別の前提（Windows は **Microsoft C++ Build Tools** の「C++ によるデスクトップ開発」ワークロード）。詳細 → <https://tauri.app/start/prerequisites/> |

> どちらの場合も、AI 機能を使うには **OpenRouter の API キー**が必要です（無料枠あり）。取得は <https://openrouter.ai/keys>。設定方法は各手順の中で説明します。

### A. インストールして使う（最短・推奨）

1. **インストーラを入手**：[Releases](https://github.com/loki-co-sudo/lokicode/releases/latest) から最新の `*.exe`（NSIS インストーラ）をダウンロード。
2. **インストール**：`.exe` を実行。スタートメニュー／デスクトップにショートカットが作成されます。
3. **起動**：アイコンをダブルクリック。
4. **API キーを設定**：左下の **⚙️（設定）** → 「OpenRouter API キー」に貼り付け → 保存。
   （キーは OS のアプリ設定フォルダに保存され、画面側には出ません。）
5. **（任意）モデルを選ぶ**：設定でモデルを選択（未選択でも動きますが、注意が表示されます）。おすすめは[最初の一歩](#最初の一歩api-キー設定とモデル選択)を参照。

以降は起動するたびにバナーで**自動アップデート**を案内します。

---

### B. ソースから動かす（開発者向け）

```bash
# 1. リポジトリを取得
git clone https://github.com/loki-co-sudo/lokicode.git
cd lokicode

# 2. 依存関係をインストール
npm install

# 3. API キーを設定（開発時は .env を使う）
cp .env.example .env
#   → .env を開き OPENROUTER_API_KEY=sk-or-... を記入

# 4. 起動（デスクトップアプリ。初回は Rust のビルドに数分かかります）
npm run tauri dev
```

- **フロントだけ確認したいとき**（Rust 不要・UI 確認用。Tauri 機能は動きません）：`npm run dev`
- **配布用インストーラを作る**：`npm run tauri build` → [開発・ビルド](#開発ビルド)参照

> パッケージ版（ダブルクリック起動）は `.env` を読みません。A の手順4と同じく、起動後に **⚙️ 設定**からキーを入力してください。
> キーの解決順は **アプリ設定ファイル > 環境変数 / `.env`** です。

## 最初の一歩（API キー設定とモデル選択）

1. **API キー**：設定（⚙️）→ OpenRouter API キー。<https://openrouter.ai/keys> で取得。
2. **モデル**：設定でモデルを選びます。一覧は OpenRouter から自動取得され、**コスト分類（無料/低/中/高/変動）・コンテキスト長・ツール対応**などで絞り込めます。
   - **おすすめ（コスパ重視）**：ディープシンクの **思考モデル＝安価で賢い軽量モデル**（例 `deepseek/deepseek-v4-flash`。無料枠で試すなら `openrouter/free`）、**合成モデル＝高性能モデル**（例 `anthropic/claude-opus-4.8` や `deepseek/deepseek-v4-pro`）。
   - 思考モデルは未設定なら自動で `openrouter/free`（既定）を使用します。コスパと品質のバランスを上げたいときは `deepseek/deepseek-v4-flash` 級の有料軽量モデルに切り替えると安定します。なお**ディープシンク使用時**は、無料ルーターのままだと品質が不安定なため注意の警告が表示されます（モデル要件ゲート・後述）。
3. **モードを選ぶ**（チャット入力上のトグル）：
   - **Agent**：AI がツールでファイル読み書き・コマンド実行まで行う（既定）。**Chat** に切り替えると会話のみ。
   - **ディープシンク**：難しい調査・設計向けに多段階で考える（後述）。速度優先なら OFF、または「エフォート」を**速度**に。
   - **エフォート（速度 / バランス / 品質）**：コスト・速度と精度のバランスを**1クリックで一括調整**。速度＝最速・最安（検証少なめ・アンサンブル無効）／バランス＝既定／品質＝最高精度（検証の合格基準を上げ、アンサンブル幅3・裏取り多め）。深さ・広さスライダーはこの上での微調整。
4. **OpenAI 互換エンドポイント**（任意）：設定の「API ベース URL」を指定すると Ollama 等のローカル LLM／他プロバイダも使えます。

> ⚠️ エージェントは実際にファイルを書き換え・コマンドを実行できます。`write_file` / `run_command` は既定で**実行前に差分プレビューつきの承認**を求めます。

## 主な機能

### AI エージェント
ファイル読み書き・コマンド実行・コードベース横断検索（grep）などのツールを使い、依頼を実際の作業として代行します。

- `write_file` / `run_command` は実行前に承認（差分プレビュー付き）。**承認レベルを3段階**から選択：
  - **慎重**：書き込み・コマンドを毎回確認
  - **標準（既定）**：編集や安全なコマンド（`git status`/`log`/`diff`、ビルド/テスト等）は自動承認し、**破壊的操作（`rm` 等）と git の変更系（`commit`/`push`/`reset`/`checkout` 等）だけ確認**
  - **全自動**：確認なしで実行
- `run_command` は **PowerShell（pwsh 優先）で非表示実行**：統合ターミナルと同じシェルで、コンソール窓を一切出さずに実行（黒い窓が点滅しない）。出力はツールカードに表示
- **承認画面で「🔍 これは何をする？」**：実行せずに、そのコマンド/変更が何をするか・副作用・リスクを AI（思考モデル）が説明。読んでから承認/拒否を判断できる
- **削除・不可逆な操作の警告**：`rm`/`del`/`Remove-Item`/`git clean`/`git reset --hard` 等を検知すると、承認画面に赤い注意を追加表示
- マルチステップ作業は **TODO 計画**として可視化、回答は**ストリーミング表示**
- 待機中は**「思考インジケータ」**（呼吸するオーブ＋巡回する文言＋経過時間）で進行中/停止中が一目で分かる
- **逆質問（`ask_user`）**：依頼が曖昧で結果を左右するとき、推測せず**1問だけ確認**してから進む
- **セルフチェック**：回答前に目的・制約に照らして自己点検し、必要なら自動修正（トグル・既定 ON）
- **忠実な編集**：追記・追加の依頼で**周囲の既存文を勝手に書き換え・言い換えしない**よう指示し、未確認の技術詳細を捏造しないよう抑制。`write_file` は**先頭の BOM（U+FEFF）を自動除去**し、Markdown 見出しの破損や差分汚染を防止
- **少ない往復で完了（効率重視の指示）**：独立した読み取り・検索は**1ターンにまとめて並列発行**し、既に文脈にある結果は**読み直さない**。シェルは関連手順を `;` で 1 コマンドに連結し、**非対話で実行**（ページャ／エディタを開かない）。git は `git --no-pager status; ... diff` のように**まとめて確認**し、`git add -A; git commit -m "..."` で**ステージとコミットを一括**。`reset --hard`／`push --force`／`clean -f` 等の**破壊的・履歴改変系は明示依頼がない限り実行しない**
- **編集チェックポイント**：このセッションの AI 変更を**ワンクリックで一括 Undo**。書き込むたびに **Git パネル／エクスプローラがリアルタイム更新**
- **実行接地の自己修正**：設定の「検証コマンド」（例 `npm run build`）を、AI がファイルを変更した後に自動実行。**失敗すれば出力を AI に渡して直させ、通るまで再試行**（最大2回・既定OFF）
- **ループモード**（トグル・既定 OFF）：上記を「変更→検証→修正」の**本格的な開発ループ**に拡張。検証コマンドが通るまで**最大5回**自動で回し、**成功時は通過した出力の末尾を証拠としてチャットに表示**。**同じエラーが2回連続**したら思考の固着と判断して即停止し、新しいスレッド（別コンテキスト）や強いモデルでの修復を提案。ループ中の AI には「テスト/アサーションの削除・弱体化・スキップで通すことの禁止」「チェック通過なしに完了を主張しない」を指示。⚠️ 失敗のたびに Agent 実行1周ぶんの API コストがかかるためオプトイン。仕様: [specs/loop-mode.md](specs/loop-mode.md)
- **オート（自動振り分け）**：依頼の難易度を4段階（trivial/agent/deep/deep-hard）で判定し チャット/Agent/ディープシンク を自動選択、難課題は深さ・広さも自動底上げ（易しい依頼を高速・低コストに）
- **推論エフォート（速度/バランス/品質）**：テスト時計算スケーリングの知見（検証器ガイドの halting 閾値・アンサンブル幅・十分性ゲート回数）に基づき、**コスト・速度・精度のトレードオフを1つのプリセットで切替**。エージェントへの運転指示（裏取りの深さ・調査の打ち切り基準）も連動。詳細は [specs/effort-presets.md](specs/effort-presets.md)
- **完遂・調査の打ち切り指示**：システム指示に「依頼が解決するまでターンを終えない（説明で止まらない）」「変更箇所を特定できたら調査をやめて着手する」を明記し、途中終了と調査のしすぎ（コスト・遅延の主因）を抑制
- **会話スレッドの複数管理**（名前付け・切替・削除）／**コスト上限アラート**／永続化はサイズ上限つきで安全（満杯でも落ちず、万一の表示エラーは**復旧画面**で回復）
- **@メンション**でワークスペースのファイルを文脈添付／**プロジェクト指示ファイル** `.lokicode/rules`(.md) を自動付与
- **並列サブタスク**（`parallel_query`）で複数調査を並行回収／エディタ右クリックの**クイックアクション**（選択範囲を 説明 / リファクタ / テスト生成）
- 全 OpenRouter モデルから選択（**コスト分類・コンテキスト長・ツール対応**、公開されていれば**知能/コーディング指数**を表示）／トークン・コスト使用量を表示

### ディープシンク（オーケストレータ型）
> **正式版（1.4.0 で beta 卒業）:** beta としていた主因（弱い/無料モデル構成での品質不安定・証拠なしドラフト・調整手段の不足）は、**モデル要件ゲート・最小接地・強い検証器＋多数決・エフォートプリセット**で解消済みです。残りのロードマップは増分的な最適化のみ（[卒業の根拠](specs/router-effort-link.md)）。日常のコーディングには Agent モード、難しい調査・設計・デバッグにはディープシンク（またはオート）を使い分けてください。

単発の API 呼び出しではなく、**設計ブリーフ（意図・成功基準・調査計画）→ 並列の接地調査 → 十分性ゲート → 証拠ベースのドラフト → 基準で採点する検証×N（合格で早期終了・低スコアは強モデルへ昇格）→ 最終合成 →（必要なら）実行**を行います。詳細は [パイプライン仕様 v2](specs/deep-reasoning-v2.md) / [エージェント設計 v3](specs/agent-v3.md) / [基盤設計](specs/architecture.md)。

> バージョンについて：パイプライン本体は **v2**（上記の多段オーケストレータ）。それを包む**難易度ルーター（オート振り分け）・強い検証器（賢さの壁の担保）・実行接地の自己修正**は **v3**（エージェント設計）で追加したもので、現在のディープシンクの挙動はこの v2＋v3 の組み合わせで動いています。

- **広さ（調査 1–5）と深さ（検証 1–16）を分離**：広さで網羅性、深さで正確性
- **コスト効率ルーティング**：安価な思考モデルが量（調査・ドラフト）、高性能モデルが要所（計画・最終）を担当
- **強い検証器＋ガードレール**：検証（採点）は**強モデル**で実行し、思考モデルが無料/弱くても「賢さの壁」を満たす（誤りの取りこぼし防止）
- **モデル要件ゲート（必要知能の警告）**：ディープシンクが Agent 単独より精度を出すための**最低限のモデル要件**を送信前に判定し、満たさない場合は警告バナーで「なぜダメか＋どうすればよいか」を表示（🛑 赤＝致命的 / ⚠️ 琥珀＝注意。オート ON 時も表示。ブロックはしない）。判定基準（[specs/model-gate.md](specs/model-gate.md)）：**合成モデル**は知能指数 35 未満・ルーター/変動系で致命的（検証器＝安全網が壁の下）、45 未満で注意、ツール非対応で注意。**思考モデル**はツール非対応で致命的（調査が接地できず Agent 単独より劣化）、無料ルーター系・知能指数 25 未満で注意
- **アンサンブル（Mixture-of-Agents）**：難課題ではドラフト/最終を複数生成→統合・選抜（**トグルで ON/OFF**。幅はエフォートに連動：速度=無効 / バランス=2 / 品質=3）
- **エフォート連動**：検証の合格点（早期終了の閾値）・強モデル昇格の閾値・十分性ゲートの回数も「エフォート」プリセットで一括調整（速度 78/60/1回 ↔ 品質 92/78/3回）
- **フェーズ反復バジェット（1.5.0・速度改善の本命）**：調査・改善などの**読み取り専用フェーズのツールループに反復上限**（速度6 / バランス10 / 品質14）。実走計測で「軽量モデルは1ターン1読みで数十回徘徊し、コンテキストが35kトークンまで膨張して遅延とコストが悪化」と判明したため。調査プロンプトにも「独立した読みは1ターンにまとめる・バジェット内で即座に構造化出力へ」を明記。実行フェーズ（本作業）は従来上限のまま。**実測効果: 同一の分析タスク（balanced・DeepSeek v4 pro/flash）で約21分→約12分（45%短縮）、品質アサーションは全維持**（e2e Run A）
- **classify ∥ brief 並列化（1.5.0）**：実行要否の判定と設計ブリーフは独立なので並列実行（合計遅延＝遅い方のみ）
- **実効レベルの事前推定（1.5.0）**：モデルの知能指数と設定（検証深さ・アンサンブル・接地・judge多数決）から、**このモデルの組み合わせがどの級の賢さで動くか**をコスト概算の隣に表示（例「指数 ~62（合成単発 55＋パイプライン +7）」）。テスト時計算の研究知見（効果は約1ティア＝指数+10前後で飽和、思考モデルが弱いと半減）に基づく**目安**で、ベンチ実測ではない。仕様: [specs/speed-and-level.md](specs/speed-and-level.md)
- **接地と検証**：実ファイル・実コマンド結果で裏取りし `file:line` を引用。検証は欠陥列挙→改善の敵対的レビュー。調査・実行フェーズには**ワークスペースルートを引き渡し**、`grep_search` の既定ルートと `run_command` の既定 cwd が正しく機能（1.4.1 で修正）
- **最小接地（幻覚対策）**：広さ1のままアンサンブルが有効なケース（従来は**証拠ゼロでツールなしドラフト**を生成する幻覚温床だった）では、ドラフト前に GOAL を副問とした読み取り専用の接地調査を1回自動実行
- **計画→実行の引き渡し**：分析パイプライン自体は読み取り専用。**実行を伴う依頼（ファイル編集・コマンド・git commit/push 等）と判定した場合**は、実行要否の判定を頑健化（1.4.1: 弱いモデルが一語指示を無視して長文で答えても**最後の YES/NO を採用**、「説明・分析・整理は NO」を明示、判定不能時は安全側＝分析パス。実走 e2e で発見した誤判定の修正）した上で、重い読み取り分析ループ（調査・ドラフト・検証）を**丸ごとスキップ**し、設計ブリーフ（計画）から直接、本物の Agent 実行パスへ引き渡して承認付きで**実際に**変更・コマンドを実行し結果を報告（大幅な高速化）。純粋な分析タスクでは従来どおり多段で深く考え、最終合成は「やった体」で書かず実行していない作業は手順として提示
  - **実行の可視化**：実行フェーズは通常の Agent 実行と同じく、**TODO チェックリスト（進捗）・途中経過のストリーミング表示・ツールカード**が出るので「いま何を・どこまで」やっているかが分かる。チェックリストは更新のたびに**最新の位置へ移動**して表示（上に戻らなくても進捗が見える）
  - **空振り防止**：実行モデルがツールを使わず説明だけで終了した場合は、実際に作業するよう自動で1回再指示（計画はシステム文脈として渡し、"もう答えた"と勘違いして即終了するのを防止）
  - **途中終了の防止（計画完了ガード）**：実行モデルが**チェックリストにまだ未完了のステップが残っているのに途中で終了**しようとした場合、その停止を拒否し、残っているステップ一覧を渡して**同じ会話の続き（それまでのツール結果を保持したまま）として自動継続**。未完了がなくなる（=全ステップ completed）まで続ける（連続の空振りは最大3回までで打ち切り、無限ループを防止）。停止を拒否した際は理由を `[execute] stop requested but N plan step(s) incomplete` として F12 コンソールに出力
  - **実行は高速モデルで**：計画は強モデル、実行（機械的なツール操作が中心）は**思考モデル（軽量・高速）**で回し、1ステップあたりの待ち時間を短縮。手数も最小化するよう指示
- **タイミング計測**：各フェーズの所要時間と LLM 呼び出し回数を F12 開発者ツールのコンソールに出力（遅い箇所の特定用）。フェーズ単位は `[deepthink]`、実行ループ（Agent）は **反復ごと**に `[execute]`/`[agent]`（各 LLM 往復・各ツール実行の秒数、承認待ち）を逐次出力
- **jank モニタ**：メインスレッドを 50ms 以上ブロックした処理を `[jank] Nms` としてコンソールに自動出力（UI が固まった原因の特定用。コンソールで `__jankWorst()` を実行すると最大ブロック時間が分かる）
- **オート（自動振り分け・難易度連動）**：依頼の難易度を1コールで **trivial / agent / deep / deep-hard の4段階**に判定し、チャット/Agent/ディープシンクを**自動で選択**（易しい依頼を高速・低コストに）。deep は深さ3・広さ2以上、**deep-hard（多ファイル横断の分析・原因不明のバグ・大規模で危険なリファクタ）は深さ5・広さ3以上＋アンサンブル強制 ON** に自動底上げ（スライダーは下限として尊重）。トグルで ON
- **検証器の多数決（品質エフォート限定）**：エフォート「品質」では検証（LLM-as-judge）を**2本並列**で実行し、スコアは低い方（保守的）・欠陥指摘は和集合を採用。採点ノイズによる早期終了の誤りを平滑化
- **送信前のコスト概算**（実使用量から自動補正）

## リリースと自動アップデート

インストール版は起動後に新バージョンを検知し、アプリ内で自己更新できます（Tauri updater + GitHub Releases）。

- **アプリ内**：起動時に自動で 1 回チェック（サイレント）。新版があれば**ウィンドウ上部のバナー**で「新しいバージョン vX.Y.Z があります」と案内し、「今すぐ更新して再起動」で適用。手動チェックはコマンドパレット（Ctrl+Shift+P）の「更新を確認」から。
- **配布サーバ**：GitHub Releases（`loki-co-sudo/lokicode`）。エンドポイントは `…/releases/latest/download/latest.json`。成果物は minisign 鍵で署名し、埋め込み公開鍵で検証。

### リリース手順（自動化）

1. `package.json` / `src-tauri/tauri.conf.json` / `src-tauri/Cargo.toml` の `version` を上げる。
2. `git tag vX.Y.Z && git push origin vX.Y.Z` を push。
3. GitHub Actions（`.github/workflows/release.yml`）が Windows でビルド・署名し、インストーラ・`.sig`・`latest.json` を添付した Release を公開。

### 必要な GitHub Secrets（リポジトリ設定で 1 回だけ登録）

| Secret | 値 |
| --- | --- |
| `TAURI_SIGNING_PRIVATE_KEY` | `~/.lokicode/updater.key` の中身（全文） |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | 鍵生成時に設定したパスワード（無しなら空） |

> 秘密鍵・パスワードはリポジトリ外で保管してください（失うと更新に署名できません）。updater を含む最初の版だけは手動インストールが必要で、それ以降が自動更新の対象です。

## セキュリティ

**鍵と通信**
- API キーは **Rust プロセス側**で保持し、フロントエンド（webview）には渡しません。`.env` は `.gitignore` 済み（`.env.example` には**実キーを入れない**）。パッケージ版のキーは OS のアプリ設定フォルダ（`openrouter.json`）に保存。
- LLM API 呼び出しはすべて Rust 側（reqwest）から行い、webview は外部へ直接通信しません。
- 自動アップデートは **minisign 署名を検証**してから適用します（HTTPS）。

**Webview の堅牢化**
- **CSP を適用**（`default-src 'self'` ベース）。Markdown は生 HTML を描画しない（`react-markdown`、`rehype-raw`/`dangerouslySetInnerHTML` 不使用）ため、AI 出力やファイル内容に混入したスクリプトは**実行されず表示されるだけ**。
- Monaco エディタは**ローカル同梱**（CDN から読み込まない）。オフラインで動作し、外部スクリプト源を持ちません。

**エージェントの実行リスクと対策**
- ⚠️ エージェントは実際にファイルを書き換え・コマンドを実行できます。`write_file` / `run_command` は既定で**実行前に承認**を求めます（承認3段階。既定の「標準」は破壊的操作と git 変更系を確認）。
- **「全自動」承認**を選ぶとすべての確認が外れます。有効化時に**警告ダイアログ**を表示します。信頼できないコード/リポジトリでは、隠れた指示（プロンプトインジェクション）で破壊的コマンドや情報の外部送信が起こりうるため、内容を理解した上でのみ使ってください。
- **ワークスペース外へのアクセス制限**（設定・**既定 ON**）：AI のファイル操作（読み取り/書き込み/一覧/grep）と `run_command` の作業ディレクトリを、開いているワークスペース内に限定します。`~/.ssh` 等の機密ファイルを AI が読んで外部モデルへ送るのを防ぎます。横断作業が必要なときは OFF にできます（影響は設定のヘルプに表示）。なお `run_command` はシェルのため、コマンド自体が外部パスを参照することは完全には防げません（制限されるのは作業ディレクトリ）。
- 既知の軽微な点：`git push/pull` の認証で GitHub トークンを git の `-c http.extraheader` 経由で渡すため、**子プロセスの引数に一瞬現れます**（共有 PC で他ユーザーがプロセス一覧で覗ける理論上のリスク）。単一ユーザーのデスクトップ利用では実害は小さいと判断し現状維持。気になる場合は資格情報ヘルパへの移行が将来の選択肢です。

## バージョン方針（SemVer）

バージョンは [Semantic Versioning](https://semver.org/lang/ja/)（`MAJOR.MINOR.PATCH`）に従います。

- **MAJOR**：後方互換を壊す変更
- **MINOR**：後方互換のある機能追加
- **PATCH**：後方互換のあるバグ修正

アプリには公開 API がないため、ここでの「互換性の契約」は次のユーザー資産を指します。**1.0 以降、これらを壊す変更は MAJOR を上げます**：

- 設定ファイル `openrouter.json`（API キー・モデル等）の形式
- 会話・キャリブレーション・最近開いた項目などの localStorage スキーマ
- プロジェクト指示 `.lokicode/rules` 等の形式
- キーバインドの既定と自動アップデート機構（`latest.json` / 署名）

`0.x` 系（現行）は SemVer 上「いつでも変更しうる」段階です。互換が崩れる場合は可能な範囲で移行処理（例：`chatStorage` の旧キー移行）を入れます。

- ✅ **検証出力の頑健化**：`parseJudgment` の解析失敗時、汎用文でなく審査の生出力を defect として渡し、REFINE に実質的な指針を与える
- ✅ **ブリーフ解析の頑健化**：`parseBrief` がセクション名の表記揺れ（OBJECTIVE/目標、成功基準、制約 等）に対応

**残りの候補（優先度順・いずれも増分的な最適化）**
- **モデル自動最適化**：モデル別の「検証スコア×実コスト」台帳を蓄積し、多腕バンディットでコスパ最良モデルを自動選択
- **構造化された証拠**：findings にソース/確度タグを付け、「どの問いの答えか」「検証済みか仮説か」を保持
- **中断の即応性**：API／ツール実行の途中までの即時停止
- **ループモードのディープ実行フェーズへの適用**：実行フェーズの計画完了ガードとの相互作用を検証の上で（[specs/loop-mode.md](specs/loop-mode.md) 将来枠）

### 新機能アイデア（今後追加したいもの）

#### AI 支援の強化

- **マルチエージェント協調実行** — 複数の AI エージェントが独立したコンテキストで並行作業（例：バックエンド API とフロントエンド画面を同時に実装、最後に統合レポート）。parallel_query を拡張し、それぞれのエージェントがツールを使いながら並行進行。
- **AI コードレビュー** — ステージされた変更やブランチ差分を AI が自動レビュー。バグ・セキュリティ・パフォーマンス・可読性の問題点を指摘し、修正案を提示。コミット前の品質ゲートとして機能。

#### IDE 機能の拡充

- **デバッガ統合（DAP: Debug Adapter Protocol）** — ブレークポイント設定・ステップ実行・変数ウォッチ・コールスタック表示。AI エージェントがデバッガを操作し、ランタイム情報を使った原因特定と修正を自動化（「このバグをデバッグして直して」で完結）。
- **テストランナー統合** — ワークスペース内のテストを自動検出し、実行・結果をツリービュー表示。失敗テストの詳細・差分表示。カバレッジ可視化。AI がテスト失敗を見て自動修正するループと連携。

#### プラットフォーム拡大

- **macOS / Linux 対応** — Tauri v2 のクロスプラットフォーム性を活かし、macOS（.dmg）と Linux（.deb / .AppImage）のビルド・配布に対応。CI に macOS と Linux ランナーを追加。

#### カスタマイズと拡張

- **エージェントプリセット** — 「Python 開発用」「コードレビュー用」「ドキュメント生成用」「リファクタリング用」など、モデル選択・ツールセット・システム指示・承認レベルをプリセット化。ワンクリックで切り替え、プロジェクト固有の .lokicode/presets でも上書き可能。

#### パフォーマンスとプライバシー

- **ローカルモデル自動連携** — Ollama / LM Studio / vLLM の起動を自動検出し、利用可能なローカルモデル一覧をモデルピッカーに統合表示。モデル能力に応じたプリセット最適化（ツール対応の有無・コンテキスト長など）を自動判定。
